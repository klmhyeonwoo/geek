# WRD: Nuxt Daily Todo Heatmap

- 작성일: 2026-04-10
- 문서 버전: v0.2
- 문서 정의: Work Requirements Document
- 연관 문서: `PRD.md`

## 1. 문서 목적

본 문서는 PRD를 실제 구현 작업으로 전환하기 위한 실행 기준 문서다. 제품 요구사항을 바로 개발 가능한 작업 단위로 분해하고, 각 단계의 산출물, 의존성, 완료 조건을 정의하는 것이 목적이다.

이 문서는 아래 항목을 명확히 한다.

- 어떤 작업을 어떤 순서로 수행할지
- 각 작업에서 무엇이 완성 상태인지
- 작업 간 선행 의존성이 무엇인지
- MVP 릴리스 전에 반드시 검증해야 할 항목이 무엇인지

## 2. 프로젝트 범위

### 2.1 구현 대상

- Nuxt 3 기반의 일일 투두 앱
- Supabase Auth 기반 이메일 회원가입/로그인/로그아웃
- 사용자별 투두 데이터 분리
- 오늘 날짜 투두 CRUD
- 날짜별 완료 수 기반 히트맵
- 과거 날짜 조회
- 오늘/주간/월간 랭킹
- 관리자 전용 회원 목록 페이지

### 2.2 구현 제외 범위

- 소셜 로그인
- 비밀번호 재설정/계정 복구
- 관리자에 의한 회원 수정/삭제/정지 처리
- 팀 협업
- 과거 날짜 편집
- 태그/카테고리/우선순위
- 알림 기능

### 2.3 작업 가정

- 인증 방식은 Supabase Auth의 이메일/비밀번호 방식을 사용한다.
- 회원가입 직후 `profiles` 레코드가 생성되어야 한다.
- 일반 사용자는 자신의 투두 데이터만 접근할 수 있다.
- 관리자 권한은 `profiles.role = 'admin'`으로 관리한다.
- 개인 투두 화면의 날짜 기준은 사용자 로컬 타임존을 사용한다.
- 랭킹 집계 기준은 서비스 타임존 `Asia/Seoul`을 사용한다.
- 주간 랭킹의 시작 요일은 월요일로 고정한다.
- 히트맵은 최근 6개월 범위를 우선 구현한다.

## 3. 최종 산출물

MVP 완료 시 아래 산출물이 존재해야 한다.

1. Nuxt 3 프로젝트 베이스 코드

3. `profiles`, `todo_days`, `todo_items` 스키마와 기본 제약 조건
4. Row Level Security 정책
5. 회원가입/로그인/로그아웃 UI
6. 오늘의 투두 입력/수정/삭제/완료 기능
7. 날짜별 히트맵 UI
8. 과거 날짜 조회 UI
9. 오늘/주간/월간 랭킹 조회 기능
10. 관리자 회원 목록 페이지
11. 실행 방법과 환경 변수 정리가 포함된 README 또는 설정 문서

## 4. 작업 스트림

### 4.1 Workstream A: 프로젝트 베이스 세팅

#### 목적

- Nuxt 앱과 Supabase 연동의 기본 골격을 실행 가능한 상태로 만든다.

#### 작업 항목

- Nuxt 3 프로젝트 초기 생성
- 패키지 매니저 및 스크립트 정리
- 기본 디렉터리 구조 정의
- 환경 변수 구조 정의
- Supabase client 초기 연결 구조 구성
- 인증 세션을 다룰 공통 유틸 또는 플러그인 구조 정의

#### 산출물

- 실행 가능한 Nuxt 앱
- `.env.example` 또는 이에 준하는 환경 변수 문서
- 기본 앱 레이아웃과 엔트리 페이지
- Supabase 연결 모듈

#### 완료 기준

- 로컬에서 앱이 정상 실행된다.
- Supabase URL/Key를 환경 변수로 분리할 수 있다.
- 인증 및 데이터 작업을 진행할 기본 구조가 준비된다.

### 4.2 Workstream B: 인증 및 프로필 기반 구축

#### 목적

- 회원가입, 로그인, 로그아웃, 세션 유지, 프로필 생성 구조를 마련한다.

#### 작업 항목

- 회원가입 화면 구현
- 로그인 화면 구현
- 로그아웃 동작 구현
- 세션 복원 및 인증 상태 감지 구현
- 보호 라우트 미들웨어 구현
- 회원가입 시 `profiles` 레코드 생성 방식 정의
- 로그인 시 `last_login_at` 갱신 방식 정의

#### 산출물

- 회원가입/로그인 UI
- 인증 상태 관리 로직
- 보호 라우트 미들웨어
- 프로필 생성/조회 흐름

#### 완료 기준

- 신규 사용자가 가입 후 로그인할 수 있다.
- 인증되지 않은 사용자는 보호된 화면에 접근할 수 없다.
- 새로고침 후에도 세션이 유지된다.
- 각 인증 사용자에 대해 프로필 레코드가 존재한다.

### 4.3 Workstream C: 데이터 모델, 권한, 집계 구조

#### 목적

- 사용자별 데이터 저장 구조와 RLS, 랭킹 집계 기반을 확정한다.

#### 작업 항목

- `profiles` 테이블 설계
- `todo_days` 테이블 설계
- `todo_items` 테이블 설계
- 유니크 제약 조건 및 FK 정의
- 기본 인덱스 검토
- `todo_days.completed_count`, `total_count` 집계 업데이트 방식 정의
- 사용자 본인 데이터 접근용 RLS 정책 정의
- 관리자 회원 목록 조회용 RLS 정책 정의
- 랭킹 조회용 SQL View 또는 RPC 설계

#### 산출물

- Supabase SQL 스키마 초안
- RLS 정책 정의서 또는 SQL
- 랭킹 집계 함수 또는 뷰 설계안

#### 완료 기준

- 동일 사용자/동일 날짜에 중복 day 레코드가 생기지 않는다.
- 일반 사용자는 자기 데이터만 접근 가능하다.
- 관리자는 회원 목록을 조회할 수 있다.
- 오늘/주간/월간 랭킹 집계에 필요한 데이터 구조가 확보된다.

### 4.4 Workstream D: 오늘의 투두 기능

#### 목적

- 로그인한 사용자가 자신의 오늘 날짜 기준으로 투두를 관리할 수 있게 한다.

#### 작업 항목

- 오늘 날짜 계산 로직 구현
- 오늘 목록 조회 로직 구현
- day 레코드 조회 또는 생성 로직 구현
- 투두 추가 기능 구현
- 투두 수정 기능 구현
- 투두 삭제 기능 구현
- 완료/미완료 토글 구현
- 완료 시 `completed_at` 반영 로직 구현
- 낙관적 업데이트 또는 저장 후 동기화 처리

#### 산출물

- 오늘의 투두 입력 컴포넌트
- 오늘의 투두 리스트 컴포넌트
- CRUD 동작에 연결된 데이터 로직

#### 완료 기준

- 로그인한 사용자가 오늘 날짜에서 항목 생성, 수정, 삭제, 완료 처리를 수행할 수 있다.
- 완료 상태 변경 시 집계값이 갱신된다.
- 새로고침 후에도 자신의 데이터가 유지된다.
- 저장 실패 시 사용자에게 오류가 표시된다.

### 4.5 Workstream E: 히트맵 및 날짜 조회 기능

#### 목적

- 날짜별 완료량을 시각화하고 과거 기록 조회 경험을 제공한다.

#### 작업 항목

- 히트맵 데이터 조회 로직 구현
- 완료 수 기준 색상 레벨 매핑 구현
- 히트맵 셀 렌더링 구현
- 선택 날짜 상태 관리 구현
- 날짜 상세 조회 패널 구현
- 오늘/과거 날짜 상태 분기 구현
- 최근 6개월 범위 계산 구현

#### 산출물

- 히트맵 컴포넌트
- 선택 날짜 상세 패널
- 날짜 클릭 시 상세 조회 흐름

#### 완료 기준

- 최근 6개월 기준 히트맵이 렌더링된다.
- 날짜 선택 시 해당 날짜의 투두 목록이 표시된다.
- 과거 날짜는 읽기 전용 상태로 조회된다.
- 완료 개수 변화가 히트맵 명도에 반영된다.

### 4.6 Workstream F: 랭킹 기능

#### 목적

- 완료량 기준으로 오늘/이번 주/이번 달 할일 부자 랭킹을 제공한다.

#### 작업 항목

- 서비스 타임존 `Asia/Seoul` 기준 기간 계산 유틸 정의
- 일간 랭킹 집계 쿼리 구현
- 주간 랭킹 집계 쿼리 구현
- 월간 랭킹 집계 쿼리 구현
- 공동 순위 처리 규칙 구현
- 랭킹 상위 목록 UI 구현
- 1위 강조 UI 구현
- 빈 상태/로딩/오류 상태 구현
- 완료/미완료 변경 후 랭킹 재조회 흐름 구현

#### 산출물

- 랭킹 조회 함수 또는 RPC 연동 코드
- 오늘/주간/월간 랭킹 컴포넌트
- 랭킹 기준 타임존 안내 문구

#### 완료 기준

- 현재 기간 기준 랭킹이 올바르게 집계된다.
- 동일 완료 수 사용자는 공동 순위로 표시된다.
- 완료 상태 변경 시 랭킹이 빠르게 다시 반영된다.
- 랭킹 데이터가 없을 때 빈 상태가 표시된다.

### 4.7 Workstream G: 관리자 회원 목록 페이지

#### 목적

- 관리자만 접근 가능한 회원 목록 페이지를 제공한다.

#### 작업 항목

- 관리자 전용 라우트 구현
- 관리자 권한 체크 로직 구현
- 회원 목록 조회 쿼리 구현
- 회원 목록 테이블 또는 리스트 UI 구현
- 기본 정렬 또는 페이지네이션 구현
- 비관리자 접근 거부 처리 구현

#### 산출물

- 관리자 전용 페이지
- 회원 목록 조회 로직
- 권한 체크 미들웨어 또는 가드

#### 완료 기준

- 관리자 계정은 회원 목록을 조회할 수 있다.
- 비관리자 계정은 관리자 페이지에 접근할 수 없다.
- 신규 가입 회원이 목록에 반영된다.

### 4.8 Workstream H: 상태 처리 및 UX 보강

#### 목적

- 인증, 투두, 랭킹, 관리자 화면 전반의 필수 상태 처리를 보완한다.

#### 작업 항목

- 인증 오류 메시지 정리
- 초기 로딩 상태 정의
- 빈 상태 UI 구현
- 저장 오류 UI 구현
- 상세 조회 오류 UI 구현
- 자정 이후 날짜 전환 처리
- 랭킹 기준 타임존 안내 문구 보강
- 모바일 반응형 레이아웃 정리

#### 산출물

- 로딩/오류/빈 상태 UI
- 날짜 전환 대응 로직
- 모바일 대응 레이아웃

#### 완료 기준

- 데이터가 없는 경우 적절한 안내 문구가 보인다.
- 인증 실패, 저장 실패, 조회 실패 시 사용자에게 상태가 전달된다.
- 모바일과 데스크톱 모두 주요 기능을 수행할 수 있다.

- 날짜 경계값 테스트

- 랭킹 집계 테스트
- 관리자 권한 테스트
- 환경 변수 설정 확인
- 실행 및 배포 가이드 정리

#### 산출물

- 테스트 체크리스트
- 실행 가이드 또는 README 업데이트

#### 완료 기준

- 핵심 수용 기준이 모두 통과한다.
- 초기 실행 방법을 문서만 보고 재현할 수 있다.

4. 오늘의 투두 CRUD 구현
5. 히트맵 및 과거 날짜 조회 구현

9. 검증 및 문서 정리

이 순서를 권장하는 이유는 사용자 인증과 권한 모델이 먼저 안정되어야 이후 투두 CRUD, 랭킹, 관리자 기능의 데이터 접근 규칙을 한 번에 정리할 수 있기 때문이다.

## 6. 작업 의존성

- 보호 라우트와 사용자별 CRUD는 인증 및 세션 관리가 먼저 준비되어야 한다.
- `todo_days`, `todo_items` 구현은 `profiles`와 RLS 정책이 먼저 정해져야 한다.
- 랭킹 구현은 `completed_at` 기록과 기간 집계 방식이 먼저 확정되어야 한다.
- 관리자 페이지는 역할 모델과 관리자 조회 정책이 먼저 필요하다.
- 자정 전환 처리는 오늘 날짜 계산 로직이 먼저 확정되어야 한다.

## 8. Definition of Done

아래 조건을 모두 만족해야 MVP 작업 완료로 본다.

- 이메일 회원가입과 로그인, 로그아웃이 정상 동작한다.
- 인증되지 않은 사용자는 보호된 화면에 접근할 수 없다.
- 일반 사용자는 자신의 투두 데이터만 조회/수정할 수 있다.
- 오늘 날짜 기준 투두 CRUD가 정상 동작한다.
- Supabase에 데이터가 저장되고 새로고침 후 유지된다.
- 최근 6개월 히트맵이 표시된다.
- 완료 항목 수에 따라 히트맵 색 농도가 달라진다.
- 과거 날짜 기록을 조회할 수 있다.
- 오늘/주간/월간 랭킹이 올바르게 표시된다.
- 관리자 페이지에서 회원 목록을 조회할 수 있다.
- 비관리자 계정은 관리자 페이지 접근이 차단된다.
- 빈 상태와 오류 상태가 최소 수준으로 처리된다.
- 모바일과 데스크톱에서 주요 기능이 동작한다.

## 9. 검증 체크리스트

### 9.1 인증 검증

- 새 사용자가 회원가입할 수 있는가
- 기존 사용자가 로그인할 수 있는가
- 로그아웃이 정상 동작하는가
- 새로고침 후에도 세션이 유지되는가
- 보호된 라우트 접근 시 로그인으로 유도되는가
- 중복 이메일 또는 잘못된 비밀번호 오류가 전달되는가

### 9.2 데이터 분리 및 투두 검증

- 로그인한 사용자가 오늘 날짜에 새 투두를 추가할 수 있는가
- 오늘 날짜 투두를 수정할 수 있는가
- 오늘 날짜 투두를 삭제할 수 있는가
- 오늘 날짜 투두 완료 체크가 가능한가
- 완료 수 변경 시 요약 수치가 즉시 반영되는가
- 완료 수 변경 시 히트맵 색이 즉시 반영되는가
- 다른 계정의 투두 데이터에 접근할 수 없도록 차단되는가

### 9.3 랭킹 검증

- 오늘 랭킹이 올바르게 계산되는가
- 이번 주 랭킹이 월요일 시작 기준으로 계산되는가
- 이번 달 랭킹이 월 단위로 계산되는가
- 완료 수가 같은 사용자가 공동 순위로 표시되는가
- 완료 상태 변경 후 랭킹이 갱신되는가
- 랭킹 데이터가 없을 때 빈 상태가 표시되는가

### 9.4 관리자 검증

- 관리자 계정이 회원 목록 페이지에 접근할 수 있는가
- 비관리자 계정은 관리자 페이지 접근이 차단되는가
- 신규 가입 회원이 회원 목록에 표시되는가
- 회원 목록 기본 정렬 또는 페이지네이션이 동작하는가

### 9.5 날짜 및 타임존 검증

- 로컬 타임존 기준 오늘 날짜가 올바르게 계산되는가
- 자정 이후 앱이 새 날짜 컨텍스트로 전환되는가
- 전날 데이터와 오늘 데이터가 분리되어 조회되는가
- 랭킹 집계가 `Asia/Seoul` 기준으로 계산되는가

### 9.6 품질 검증

- 모바일에서 입력과 조회가 가능한가
- 주요 버튼과 체크박스가 접근 가능한가
- 저장 실패 시 사용자에게 오류가 전달되는가
- 인증 실패 시 사용자에게 오류가 전달되는가

## 10. 리스크 및 대응

- 인증과 프로필 생성 간 불일치
  - 대응: 회원가입 직후 프로필 생성 실패를 감지하고 재시도 또는 보정 흐름을 둔다.
- RLS 정책 오류로 인한 데이터 노출
  - 대응: 사용자 계정 2개 이상으로 교차 접근 테스트를 반드시 수행한다.
- 랭킹 타임존 혼동
  - 대응: 랭킹 기준 타임존을 UI에 명시하고 집계 로직을 `completed_at` 기준으로 통일한다.
- 집계값 불일치
  - 대응: 완료 토글 시 day 집계 갱신 로직을 일관되게 두고 필요 시 재계산 함수를 둔다.
- 관리자 목록 조회 성능 저하
  - 대응: 기본 정렬/페이지네이션을 적용하고 필요한 인덱스를 검토한다.

## 11. 확정 구현 원칙

- 인증은 MVP 범위에 포함한다.
- 인증 방식은 이메일/비밀번호로 제한한다.
- 랭킹은 상위 목록과 1위 강조를 함께 제공한다.
- 랭킹 기간 기준은 일간/주간/월간 현재 진행 중 구간이다.
- 주간 시작일은 월요일이다.
- 관리자 권한은 초기에는 DB에서 수동 부여한다.
