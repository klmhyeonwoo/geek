# PRD: Nuxt Daily Todo Heatmap

- 작성일: 2026-04-10
- 문서 버전: v0.2
- 문서 정의: Product Requirements Document
- 작성 목적: Nuxt 기반의 일일 투두 관리 프로젝트 요구사항 정의

## 1. 프로젝트 개요

이 프로젝트는 사용자가 계정을 생성하고 로그인한 뒤, 매일 오늘의 할 일을 작성하고 완료 현황을 기록하며, 날짜별 생산성 흐름을 GitHub 커밋 히트맵과 유사한 블록 UI로 확인할 수 있는 일일 투두 앱을 만드는 것을 목표로 한다. 데이터 저장, 사용자 인증, 랭킹 집계는 모두 Supabase를 기반으로 구성한다.

본 서비스는 개인 생산성 관리와 가벼운 경쟁 요소를 함께 제공한다. 사용자는 자신의 오늘 할 일을 관리하는 동시에, 다른 사용자와의 비교를 통해 오늘의 할일 부자, 이번 주 할일 부자, 이번 달 할일 부자를 확인할 수 있어야 한다. 운영자는 관리자 페이지에서 회원 목록을 확인할 수 있어야 한다.

핵심 가치는 아래 4가지다.

1. 매일의 할 일을 빠르게 기록할 수 있어야 한다.
2. 완료한 일의 양이 시각적으로 누적되어 동기부여를 제공해야 한다.
3. 계정 기반으로 개인 데이터가 안전하게 분리되고 유지되어야 한다.
4. 일간/주간/월간 랭킹을 통해 꾸준함을 경쟁적으로 즐길 수 있어야 한다.

## 2. 문제 정의

기존 투두 앱은 오늘 해야 할 일 관리에는 강하지만, 계정 기반 데이터 보존, 개인 성취 시각화, 경쟁형 동기부여, 운영 관점의 회원 관리가 동시에 제공되는 경우는 드물다. 특히 공개 서비스 형태로 운영하려면 인증, 데이터 분리, 관리자 기능이 함께 정의되어야 한다.

따라서 본 프로젝트는 다음 문제를 해결해야 한다.

- 사용자가 회원가입과 로그인 후 자신의 투두 기록을 안전하게 관리할 수 있어야 한다.
- 사용자별 데이터가 명확히 분리되어 다른 사람의 투두를 볼 수 없어야 한다.
- 오늘 완료한 항목 수가 날짜별 블록의 색 농도로 반영되어야 한다.
- 특정 날짜의 투두 기록을 나중에 다시 열람할 수 있어야 한다.
- 완료량 기준으로 일간/주간/월간 랭킹을 집계해 동기부여를 제공해야 한다.
- 관리자가 회원 목록을 조회하고 서비스 운영 상태를 파악할 수 있어야 한다.

## 3. 목표

### 3.1 프로젝트 목표

- 이메일 기반 회원가입, 로그인, 로그아웃이 가능할 것
- 사용자별로 투두 데이터가 분리되어 안전하게 저장/조회될 것
- 날짜별 투두 작성과 완료 체크가 직관적으로 가능할 것
- 하루 단위 완료량이 히트맵으로 시각화될 것
- 오늘/주간/월간 랭킹이 자동 집계될 것
- 관리자 전용 회원 목록 페이지가 제공될 것

### 3.2 MVP 성공 기준

- 신규 사용자가 이메일과 비밀번호로 회원가입할 수 있다.
- 기존 사용자가 로그인 후 자신의 오늘 투두를 추가, 수정, 삭제, 완료 처리할 수 있다.
- 새로고침 후에도 세션이 유지되고 자신의 데이터가 다시 로드된다.
- 과거 날짜를 선택하면 해당 날짜의 투두 목록을 조회할 수 있다.
- 히트맵에서 완료 항목 수에 따라 블록 색 농도가 달라진다.
- 오늘/이번 주/이번 달 랭킹이 Supabase 데이터 기준으로 노출된다.
- 관리자 계정으로 로그인하면 회원 목록 페이지를 조회할 수 있다.

### 3.3 비목표

- 소셜 로그인
- 비밀번호 재설정/계정 복구 플로우
- 팀 협업 기능
- 알림/푸시 기능
- 복잡한 프로젝트/태그/카테고리 구조
- 서브태스크, 반복 일정, 캘린더 연동
- 관리자에 의한 회원 수정/정지/삭제 기능

## 4. 대상 사용자

- 매일 해야 할 일을 간단하게 기록하고 싶은 개인 사용자
- 하루의 성취도를 시각적으로 확인하고 싶은 사용자
- 다른 사용자와의 비교를 통해 동기부여를 얻고 싶은 사용자
- 가입 회원 현황을 확인해야 하는 서비스 운영자 또는 관리자

## 5. 핵심 사용자 시나리오

### 시나리오 A: 회원가입 및 로그인

1. 사용자는 회원가입 화면에서 이메일, 비밀번호, 표시 이름을 입력한다.
2. 계정 생성이 완료되면 로그인 상태로 앱에 진입하거나 로그인 화면으로 이동한다.
3. 이후 사용자는 이메일과 비밀번호로 로그인한다.
4. 로그인 후 자신의 오늘 날짜 기준 투두 리스트 화면을 본다.

### 시나리오 B: 오늘의 할 일 기록

1. 사용자는 메인 화면에서 오늘 날짜 기준의 투두 리스트를 본다.
2. 할 일을 여러 개 추가한다.
3. 완료한 항목을 체크한다.
4. 변경사항은 Supabase에 자동 저장된다.
5. 사용자는 새로고침 후에도 동일한 계정의 데이터를 확인한다.

### 시나리오 C: 히트맵으로 누적 성과 확인

1. 사용자는 메인 화면에서 날짜별 히트맵을 본다.
2. 완료 항목 수가 많은 날짜일수록 더 진한 색 블록을 확인한다.
3. 연속 기록 여부와 생산성 흐름을 시각적으로 파악한다.

### 시나리오 D: 랭킹으로 경쟁 상태 확인

1. 사용자는 메인 화면 또는 랭킹 섹션에서 오늘/이번 주/이번 달 랭킹을 본다.
2. 각 기간별 상위 사용자와 완료 개수를 확인한다.
3. 자신의 순위를 확인하며 동기부여를 얻는다.

### 시나리오 E: 관리자 회원 목록 조회

1. 관리자는 관리자 권한 계정으로 로그인한다.
2. 관리자 페이지로 이동한다.
3. 가입한 회원 목록과 기본 정보를 확인한다.
4. 비관리자 사용자는 해당 페이지에 접근할 수 없다.

## 6. 기능 요구사항

### 6.1 회원가입 및 인증

- 사용자는 이메일과 비밀번호를 사용해 회원가입할 수 있어야 한다.
- 회원가입 시 최소 아래 정보를 입력해야 한다.
  - 이메일
  - 비밀번호
  - 표시 이름
- 사용자는 이메일과 비밀번호를 사용해 로그인할 수 있어야 한다.
- 로그인한 사용자는 명시적으로 로그아웃할 수 있어야 한다.
- 새로고침 후에도 로그인 세션이 유지되어야 한다.
- 인증되지 않은 사용자가 보호된 화면에 접근하면 로그인 화면으로 유도되어야 한다.
- 중복 이메일, 잘못된 비밀번호, 유효하지 않은 입력값에 대해 사용자에게 오류 상태를 안내해야 한다.

### 6.2 사용자 프로필 및 권한

- 모든 인증 사용자는 애플리케이션 프로필 레코드를 가져야 한다.
- 프로필은 최소 아래 정보를 포함해야 한다.
  - 사용자 ID
  - 이메일
  - 표시 이름
  - 역할(`user`, `admin`)
  - 상태(`active` 기본)
  - 가입일
  - 마지막 로그인 시각
- 기본 역할은 `user`여야 한다.
- 관리자 권한은 운영자가 Supabase에서 별도로 부여할 수 있어야 한다.
- 랭킹에는 표시 이름을 사용해야 하며, 이메일은 공개하지 않아야 한다.

### 6.3 오늘의 투두 리스트

- 사용자는 자신의 오늘 날짜 기준 투두 항목을 생성할 수 있어야 한다.
- 각 항목은 최소 아래 정보를 가진다.
  - 텍스트 내용
  - 완료 여부
  - 생성 시간
  - 정렬 순서
- 사용자는 자신의 오늘 날짜 항목을 수정할 수 있어야 한다.
- 사용자는 자신의 오늘 날짜 항목을 삭제할 수 있어야 한다.
- 사용자는 자신의 오늘 날짜 항목을 완료/미완료 처리할 수 있어야 한다.

### 6.4 데이터 분리 및 접근 제어

- 사용자는 자신의 투두 데이터만 조회/수정/삭제할 수 있어야 한다.
- 다른 사용자의 투두 데이터는 직접 조회할 수 없어야 한다.
- 모든 투두 데이터는 인증된 사용자 ID와 연결되어야 한다.
- 관리자 페이지를 제외한 일반 데이터 접근은 Row Level Security를 통해 사용자 단위로 제한되어야 한다.
- 관리자 페이지의 회원 목록 조회는 관리자 역할 사용자만 가능해야 한다.

### 6.5 날짜 및 집계 기준

- 개인 투두 화면의 오늘 날짜는 사용자의 로컬 타임존 기준으로 계산한다.
- 날짜가 00:00을 지나면 오늘 리스트는 새로운 날짜 컨텍스트로 전환되어야 한다.
- 새 날짜에는 기존 날짜의 투두가 자동 복사되지 않는다.
- 새 날짜에 작성된 데이터는 이전 날짜와 별도로 저장된다.
- 랭킹 집계는 사용자별 로컬 날짜가 아니라 서비스 기준 타임존 `Asia/Seoul`로 통일해 계산해야 한다.
- 랭킹 섹션에는 집계 기준 시간이 `Asia/Seoul`임을 안내해야 한다.

### 6.6 이전 날짜 조회

- 사용자는 과거 날짜를 선택해 자신의 해당 날짜 투두 목록을 조회할 수 있어야 한다.
- 과거 날짜에 데이터가 없으면 빈 상태 메시지를 노출해야 한다.
- MVP 기준으로 과거 날짜는 읽기 전용으로 제공한다.

### 6.7 히트맵 시각화

- 날짜별 블록 UI는 GitHub 커밋 히트맵과 유사한 형태여야 한다.
- 각 날짜 블록의 색 농도는 해당 날짜의 완료된 투두 개수에 따라 달라져야 한다.
- 완료 수가 0개인 날짜는 기본색 또는 가장 연한 색으로 표시한다.
- 사용자가 블록에 마우스를 올리거나 탭하면 날짜와 완료 개수를 볼 수 있어야 한다.
- 사용자가 블록을 클릭하면 해당 날짜 상세 목록을 조회할 수 있어야 한다.

### 6.8 히트맵 색상 단계

MVP는 5단계 명도 체계를 사용한다.

- Level 0: 완료 0개
- Level 1: 완료 1개
- Level 2: 완료 2-3개
- Level 3: 완료 4-5개
- Level 4: 완료 6개 이상

추후 운영 과정에서 임계값은 조정 가능해야 한다.

### 6.9 랭킹 기능

- 랭킹은 아래 3개 구간을 제공해야 한다.
  - 오늘의 할일 부자
  - 이번 주 할일 부자
  - 이번 달 할일 부자
- 랭킹 점수는 각 기간 내 완료된 투두 항목 수를 기준으로 계산해야 한다.
- 랭킹 집계 기준 데이터는 `todo_items.completed_at`을 사용해야 한다.
- 일간 랭킹은 당일 00:00부터 23:59:59까지의 완료 건수를 사용해야 한다.
- 주간 랭킹은 `Asia/Seoul` 기준 월요일 00:00부터 현재 시점까지의 완료 건수를 사용해야 한다.
- 월간 랭킹은 해당 월 1일 00:00부터 현재 시점까지의 완료 건수를 사용해야 한다.
- 동일 완료 수 사용자는 공동 순위를 가져야 한다.
- 각 랭킹은 최소 1위 사용자와 상위 목록을 표시할 수 있어야 한다.
- 랭킹에는 표시 이름과 완료 개수만 노출해야 한다.
- 랭킹 데이터가 없으면 빈 상태를 노출해야 한다.
- 사용자의 완료/미완료 변경은 랭킹에 빠르게 반영되어야 한다.

### 6.10 관리자 페이지

- 관리자 전용 회원 목록 페이지가 제공되어야 한다.
- 페이지에는 최소 아래 정보를 표시해야 한다.
  - 표시 이름
  - 이메일
  - 역할
  - 상태
  - 가입일
  - 마지막 로그인 시각
- 회원 수가 많아질 경우를 고려해 기본적인 정렬 또는 페이지네이션이 가능해야 한다.
- 비관리자 사용자가 접근할 경우 접근 거부 또는 메인 화면 리다이렉트가 이루어져야 한다.

### 6.11 저장 및 동기화

- 회원 정보, 투두 데이터, 랭킹 집계에 필요한 원천 데이터는 Supabase에 저장되어야 한다.
- 사용자의 생성/수정/삭제/완료 처리 결과는 저장 후 즉시 UI에 반영되어야 한다.
- 조회 실패 또는 저장 실패 시 사용자에게 오류 상태를 안내해야 한다.

## 7. 화면 구성

### 7.1 인증 화면

- 회원가입 폼
- 로그인 폼
- 인증 오류 메시지 영역

### 7.2 메인 화면

- 사용자 인사 또는 표시 이름 영역
- 로그아웃 버튼
- 오늘 날짜 헤더
- 오늘의 투두 입력 영역
- 오늘의 투두 목록
- 완료 개수 요약
- 오늘/주간/월간 랭킹 섹션
- 히트맵 영역
- 선택한 날짜의 상세 보기 패널 또는 섹션

### 7.3 날짜 상세 보기

- 선택한 날짜 표시
- 해당 날짜의 투두 목록
- 각 항목의 완료 여부
- 데이터가 없는 경우 빈 상태 메시지

### 7.4 관리자 페이지

- 회원 목록 테이블 또는 리스트
- 역할/상태/가입일 등 기본 컬럼
- 페이지네이션 또는 정렬 UI
- 권한 없을 때 접근 제한 처리

## 8. 데이터 요구사항

### 8.1 권장 데이터 모델

MVP에서는 Supabase Auth와 별도 프로필 테이블, 일자 단위 데이터, 항목 단위 데이터를 함께 사용하는 구조를 권장한다.

### `auth.users`

- Supabase Auth가 관리하는 기본 인증 사용자 테이블
- 이메일/비밀번호 인증 기준 계정 원본 데이터 저장

### `profiles`

- `id`: uuid, PK, `auth.users.id` 참조
- `email`: text, not null, unique
- `display_name`: text, not null, unique
- `role`: text, not null, default `user`
- `status`: text, not null, default `active`
- `last_login_at`: timestamptz, nullable
- `created_at`: timestamptz
- `updated_at`: timestamptz

제약 조건:

- `role`은 `user`, `admin` 중 하나여야 한다.
- `status`는 최소 `active` 상태를 지원해야 한다.

### `todo_days`

- `id`: uuid, PK
- `user_id`: uuid, not null, FK -> `profiles.id`
- `target_date`: date, not null
- `timezone`: text, not null
- `completed_count`: int, default 0
- `total_count`: int, default 0
- `created_at`: timestamptz
- `updated_at`: timestamptz

제약 조건:

- `user_id + target_date` 조합은 유니크해야 한다.

### `todo_items`

- `id`: uuid, PK
- `day_id`: uuid, FK -> `todo_days.id`
- `content`: text, not null
- `is_completed`: boolean, default false
- `display_order`: int, default 0
- `completed_at`: timestamptz, nullable
- `created_at`: timestamptz
- `updated_at`: timestamptz

### 8.2 데이터 계산 규칙

- 히트맵 색상 계산 기준 값은 `todo_days.completed_count`
- `completed_count`와 `total_count`는 항목 변경 시 함께 갱신하거나, 조회 시 집계할 수 있다.
- MVP에서는 조회 성능과 구현 단순성을 위해 `todo_days`에 집계값을 캐시하는 방식을 권장한다.
- 랭킹 집계 기준 값은 `todo_items.completed_at`이며, 완료 취소 시 집계에서 제외되어야 한다.

### 8.3 접근 제어 및 공개 범위

- `profiles`는 본인 읽기/수정과 관리자 전체 조회를 지원해야 한다.
- `todo_days`, `todo_items`는 본인 소유 데이터에 한해 CRUD가 가능해야 한다.
- 랭킹 조회는 인증 사용자에게 허용하되, 표시 이름과 완료 수만 노출해야 한다.
- 관리자 페이지는 관리자 역할 사용자에게만 허용해야 한다.

### 8.4 랭킹 집계 구현 방향

- Supabase SQL View 또는 RPC 함수를 사용해 일간/주간/월간 랭킹을 계산하는 방식을 권장한다.
- 주기성 배치가 없어도 현재 기간 기준 랭킹을 조회할 수 있어야 한다.
- 필요 시 추후 Materialized View 또는 캐시 전략으로 확장 가능해야 한다.

### 9.2 백엔드/DB

- Supabase Postgres
- Supabase Auth
- Supabase client를 통한 CRUD
- Row Level Security 적용
- 랭킹용 SQL View 또는 RPC 함수

### 9.3 인증 및 보안 가정

- 모든 주요 화면은 로그인 이후 접근을 기본으로 한다.
- 브라우저 클라이언트에는 Supabase service role key를 노출하지 않는다.
- 관리자 기능은 역할 기반 제어와 RLS를 통해 보호한다.

## 10. 비기능 요구사항

- 모바일과 데스크톱에서 모두 사용 가능해야 한다.
- 새로고침 후에도 로그인 세션과 사용자 데이터가 유지되어야 한다.
- 주요 상호작용은 1초 내 체감 응답을 목표로 한다.
- 랭킹 조회는 일반적인 사용량 기준에서 2초 내 표시를 목표로 한다.
- 접근성을 위해 체크박스, 버튼, 날짜 정보, 인증 입력 요소에 명확한 레이블이 있어야 한다.
- 타임존 차이로 날짜가 잘못 저장되지 않도록 개인 기록 기준과 랭킹 집계 기준을 명확히 분리해야 한다.
- 랭킹에서는 이메일 등 민감한 회원 정보를 직접 노출하지 않아야 한다.

## 11. 예외 및 엣지 케이스

- 오늘 날짜에 투두가 하나도 없는 경우 빈 상태를 표시해야 한다.
- 과거 날짜 데이터가 없는 경우 "기록 없음" 상태를 보여줘야 한다.
- 사용자가 자정 직전에 앱을 열어둔 경우 날짜 전환 후 오늘 컨텍스트를 재평가해야 한다.
- 중복 이메일 또는 중복 표시 이름으로 회원가입을 시도하면 실패 사유를 안내해야 한다.
- 로그인 세션이 만료된 경우 보호된 화면에서 다시 로그인하도록 유도해야 한다.
- 비관리자 사용자가 관리자 페이지에 접근하면 권한 오류 또는 리다이렉트를 수행해야 한다.
- 랭킹 데이터가 전혀 없는 경우 빈 상태를 보여줘야 한다.
- 동일 완료 수 사용자가 여러 명이면 공동 1위가 발생할 수 있다.
- 개인 화면의 오늘 날짜와 랭킹 기준 날짜가 다를 수 있으므로, 랭킹 기준 타임존 안내가 필요하다.
