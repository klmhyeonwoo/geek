# PRD: Nuxt Daily Todo Heatmap

- 작성일: 2026-04-10
- 문서 버전: v0.1
- 작성 목적: Nuxt 기반의 일일 투두 관리 서비스 MVP 요구사항 정의

## 1. 제품 개요

이 제품은 사용자가 매일 오늘의 할 일을 작성하고, 완료 현황을 기록하며, 날짜별 생산성 흐름을 GitHub 커밋 히트맵과 유사한 블록 UI로 확인할 수 있는 일일 투두 서비스다. 데이터는 Supabase에 저장되며, 사용자는 오늘의 할 일을 관리하는 동시에 과거 날짜의 기록을 조회할 수 있어야 한다.

핵심 가치는 아래 3가지다.

1. 매일의 할 일을 빠르게 기록할 수 있어야 한다.
2. 완료한 일의 양이 시각적으로 누적되어 동기부여를 제공해야 한다.
3. 과거 기록을 쉽게 되돌아볼 수 있어야 한다.

## 2. 문제 정의

기존 투두 앱은 오늘 해야 할 일 관리에는 강하지만, "매일 꾸준히 기록했다"는 흐름을 직관적으로 보여주는 경험은 약한 편이다. 사용자는 하루 단위의 할 일 기록과 누적된 성취 시각화를 함께 원한다.

따라서 본 서비스는 다음 문제를 해결해야 한다.

- 사용자가 매일 새로운 할 일 목록을 작성할 수 있어야 한다.
- 오늘 완료한 항목 수가 날짜별 블록의 색 농도로 반영되어야 한다.
- 특정 날짜의 투두 기록을 나중에 다시 열람할 수 있어야 한다.

## 3. 목표

### 3.1 제품 목표

- 날짜별 투두 작성과 완료 체크가 직관적으로 가능할 것
- 하루 단위 완료량이 히트맵으로 시각화될 것
- Supabase를 통해 데이터가 안정적으로 저장되고 조회될 것
- 오늘 날짜와 과거 날짜의 기록이 명확히 분리될 것

### 3.2 MVP 성공 기준

- 사용자가 오늘 날짜에 투두를 추가, 수정, 삭제, 완료 처리할 수 있다.
- 날짜가 바뀌면 새로운 일일 리스트가 생성되거나 빈 상태로 시작된다.
- 과거 날짜를 선택하면 해당 날짜의 투두 목록을 조회할 수 있다.
- 히트맵에서 완료 항목 수에 따라 블록 색 농도가 달라진다.

### 3.3 비목표

- 팀 협업 기능
- 알림/푸시 기능
- 복잡한 프로젝트/태그/카테고리 구조
- 서브태스크, 반복 일정, 캘린더 연동

## 4. 대상 사용자

- 매일 해야 할 일을 간단하게 기록하고 싶은 개인 사용자
- 하루의 성취도를 시각적으로 확인하고 싶은 사용자
- 습관 추적과 투두 관리를 한 화면에서 보고 싶은 사용자

## 5. 핵심 사용자 시나리오

### 시나리오 A: 오늘의 할 일 기록

1. 사용자는 앱에 접속한다.
2. 오늘 날짜 기준의 투두 리스트 화면을 본다.
3. 할 일을 여러 개 추가한다.
4. 완료한 항목을 체크한다.
5. 변경사항은 Supabase에 자동 저장된다.

### 시나리오 B: 히트맵으로 누적 성과 확인

1. 사용자는 메인 화면에서 날짜별 히트맵을 본다.
2. 완료 항목 수가 많은 날짜일수록 더 진한 색 블록을 확인한다.
3. 연속 기록 여부를 시각적으로 파악한다.

### 시나리오 C: 이전 날짜 기록 조회

1. 사용자는 히트맵 또는 날짜 선택 UI에서 과거 날짜를 클릭한다.
2. 해당 날짜에 작성된 투두 목록과 완료 상태를 확인한다.
3. MVP에서는 과거 날짜는 조회 중심으로 제공한다.

## 6. 기능 요구사항

### 6.1 오늘의 투두 리스트

- 사용자는 오늘 날짜 기준의 투두 항목을 생성할 수 있어야 한다.
- 각 항목은 최소 아래 정보를 가진다.
  - 텍스트 내용
  - 완료 여부
  - 생성 시간
  - 정렬 순서
- 사용자는 오늘 날짜의 항목을 수정할 수 있어야 한다.
- 사용자는 오늘 날짜의 항목을 삭제할 수 있어야 한다.
- 사용자는 오늘 날짜의 항목을 완료/미완료 처리할 수 있어야 한다.

### 6.2 날짜 갱신 규칙

- 앱은 사용자의 로컬 타임존 기준 날짜를 사용한다.
- 날짜가 00:00을 지나면 오늘 리스트는 새로운 날짜 컨텍스트로 전환되어야 한다.
- 새 날짜에는 기존 날짜의 투두가 자동 복사되지 않는다.
- 새 날짜에 작성된 데이터는 이전 날짜와 별도로 저장된다.

### 6.3 이전 날짜 조회

- 사용자는 과거 날짜를 선택해 해당 날짜의 투두 목록을 조회할 수 있어야 한다.
- 과거 날짜에 데이터가 없으면 빈 상태 메시지를 노출해야 한다.
- MVP 기준으로 과거 날짜는 읽기 전용으로 제공한다.

### 6.4 히트맵 시각화

- 날짜별 블록 UI는 GitHub 커밋 히트맵과 유사한 형태여야 한다.
- 각 날짜 블록의 색 농도는 해당 날짜의 "완료된 투두 개수"에 따라 달라져야 한다.
- 완료 수가 0개인 날짜는 기본색 또는 가장 연한 색으로 표시한다.
- 사용자가 블록에 마우스를 올리거나 탭하면 날짜와 완료 개수를 볼 수 있어야 한다.
- 사용자가 블록을 클릭하면 해당 날짜 상세 목록을 조회할 수 있어야 한다.

### 6.5 히트맵 색상 단계

MVP는 5단계 명도 체계를 사용한다.

- Level 0: 완료 0개
- Level 1: 완료 1개
- Level 2: 완료 2-3개
- Level 3: 완료 4-5개
- Level 4: 완료 6개 이상

추후 운영 과정에서 임계값은 조정 가능해야 한다.

### 6.6 저장 및 동기화

- 모든 투두 데이터는 Supabase에 저장되어야 한다.
- 사용자의 생성/수정/삭제/완료 처리 결과는 저장 후 즉시 UI에 반영되어야 한다.
- 조회 실패 또는 저장 실패 시 사용자에게 오류 상태를 안내해야 한다.

## 7. 화면 구성

### 7.1 메인 화면

- 오늘 날짜 헤더
- 오늘의 투두 입력 영역
- 오늘의 투두 목록

- 데이터가 없는 경우 빈 상태 메시지

## 8. 데이터 요구사항

### 8.1 권장 데이터 모델

MVP에서는 일자 단위와 항목 단위를 분리한 2개 테이블 구조를 권장한다.

### `todo_days`

- `id`: uuid, PK
- `user_id`: uuid, nullable 또는 auth 연동 시 FK
- `target_date`: date, not null
- `timezone`: text, not null
- `completed_count`: int, default 0
- `total_count`: int, default 0
- `created_at`: timestamptz
- `updated_at`: timestamptz

제약 조건:

- `user_id + target_date` 조합은 유니크해야 한다.

### `todo_items`

- `id`: uuid, PK
- `day_id`: uuid, FK -> `todo_days.id`
- `content`: text, not null
- `is_completed`: boolean, default false
- `display_order`: int, default 0
- `completed_at`: timestamptz, nullable
- `created_at`: timestamptz
- `updated_at`: timestamptz

### 8.2 데이터 계산 규칙

- 히트맵 색상 계산 기준 값은 `completed_count`
- `completed_count`와 `total_count`는 항목 변경 시 함께 갱신하거나, 조회 시 집계할 수 있다.
- MVP에서는 조회 성능과 구현 단순성을 위해 `todo_days`에 집계값을 캐시하는 방식을 권장한다.

## 9. 기술 범위

### 9.1 프론트엔드

- Nuxt 3 기반
- 날짜별 투두 조회/편집 UI
- 히트맵 컴포넌트
- 로컬 타임존 기준 날짜 처리

### 9.2 백엔드/DB

- Supabase Postgres
- Supabase client를 통한 CRUD
- 필요 시 Row Level Security 적용

### 9.3 인증 가정

- 본 MVP PRD는 인증이 없는 단일 사용자 또는 개인용 배포를 우선 가정한다.
- 공개 서비스로 확장할 경우 Supabase Auth를 추가하고 사용자별 데이터 분리가 필요하다.

## 10. 비기능 요구사항

- 모바일과 데스크톱에서 모두 사용 가능해야 한다.
- 새로고침 후에도 데이터가 유지되어야 한다.
- 주요 상호작용은 1초 내 체감 응답을 목표로 한다.
- 접근성을 위해 체크박스, 버튼, 날짜 정보에 명확한 레이블이 있어야 한다.
- 타임존 차이로 날짜가 잘못 저장되지 않도록 날짜 계산 규칙이 일관되어야 한다.

## 11. 예외 및 엣지 케이스

- 오늘 날짜에 투두가 하나도 없는 경우 빈 상태를 표시해야 한다.
- 과거 날짜 데이터가 없는 경우 "기록 없음" 상태를 보여줘야 한다.
- 사용자가 자정 직전에 앱을 열어둔 경우 날짜 전환 후 오늘 컨텍스트를 재평가해야 한다.
- 완료 후 다시 미완료로 되돌리면 히트맵 집계값도 즉시 반영되어야 한다.
- 동일 날짜에 중복 `todo_days` 레코드가 생기지 않도록 보호해야 한다.

## 12. MVP 릴리스 범위

릴리스 1에서는 아래 범위를 포함한다.

- 오늘 날짜 투두 CRUD
- 완료 체크/해제
- Supabase 영속 저장
- 날짜별 히트맵 시각화
- 과거 날짜 조회

릴리스 1에서는 아래 범위를 제외한다.

- 사용자 인증
- 다중 사용자 지원
- 과거 날짜 편집
- 태그/우선순위/카테고리
- 알림 기능

## 13. 수용 기준

### AC-1 오늘 투두 작성

- 사용자가 오늘 화면에서 새 항목을 추가하면 저장 후 목록에 즉시 표시된다.

### AC-2 완료 수 반영

- 사용자가 항목을 완료 처리하면 해당 날짜의 완료 수가 증가하고 히트맵 명도가 변경된다.

### AC-3 날짜별 분리 저장

- 2026-04-10에 작성한 데이터와 2026-04-11에 작성한 데이터는 서로 다른 날짜 기록으로 조회된다.

### AC-4 과거 기록 조회

- 사용자가 히트맵에서 특정 날짜를 선택하면 해당 날짜의 투두 목록이 표시된다.

### AC-5 빈 상태 처리

- 선택한 날짜에 데이터가 없으면 빈 상태 UI가 노출된다.
